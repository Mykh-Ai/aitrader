# Стратегія Ші v1.0 — Backtesting_Spec_v0.1

**Status:** Implemented baseline contract (v0.1)  
**Phase:** 3 — Implemented baseline  
**Purpose:** Formal backtesting contract aligned to current implementation

## 1. Purpose

Phase 3 defines the formal contract for:

- deterministic ruleset formalization
- historical replay
- trade simulation
- trade-level metrics
- statistical edge validation
- robustness testing
- promotion gating toward execution design

This document exists to prevent the Backtester from inheriting hidden assumptions, silent execution semantics, or lookahead leakage.

Phase 3 begins only after Phase 2 research outputs already exist.

### 1.1 Implemented baseline status (repo truth)

The current repository contains an implemented baseline stack for all core Phase 3 modules:

- `backtester/rulesets.py`
- `backtester/engine.py`
- `backtester/ledger.py`
- `backtester/metrics.py`
- `backtester/validation.py`
- `backtester/robustness.py`
- `backtester/promotion.py`
- `backtester/orchestrator.py`

This spec keeps the contract shape and explicitly marks which parts are baseline-implemented,
which are provisional heuristics, and which remain future hardening work.

### Out of scope for this document
Phase 3 is **not**:

- live execution
- exchange integration
- order routing
- margin workflow handling
- live risk enforcement

## 2. Boundary with Analyzer / Phase 2

The Analyzer is a **facts engine + research pipeline**.

It already:

- computes derived features from raw 1m feed
- detects structural events
- extracts setup candidates
- computes forward-looking setup outcomes
- builds grouped reports, context reports, rankings, selections, shortlist, shortlist explanations, and final research summary

The Analyzer does **not**:

- perform backtesting
- perform ruleset validation
- perform statistical acceptance of trade logic
- make live execution decisions
- size positions or place orders

Phase 3 starts when a research candidate is converted into a **deterministic ruleset** and replayed as a tradeable hypothesis.

### Phase 3 owns

- ruleset formalization
- replay semantics
- cost model semantics
- trade simulation
- trade ledger generation
- trade-level metrics
- validation
- robustness testing
- promotion decisions

### Phase 3 does not own

- live signal generation
- order routing
- exchange integration
- borrow / repay workflows
- live risk controls

Those remain outside Phase 3 and belong to future execution design.

## 3. Inputs from Analyzer

Phase 3 consumes Phase 1 and Phase 2 artifacts.

### 3.1 Core required inputs

Minimum required inputs for replay-capable Phase 3 core:

- raw 1m feed from Aggregator
- Analyzer feature table
- Analyzer setup table

These are sufficient to begin ruleset replay once Phase 3 rules are formalized.

### 3.2 Optional research lineage inputs

The following are not required for basic replay, but are strongly recommended for research traceability and candidate selection:

- Analyzer event table
- Analyzer outcome table
- Analyzer reports
- Analyzer context reports
- Analyzer rankings
- Analyzer selections
- Analyzer shortlist
- Analyzer shortlist explanations
- Analyzer research summary

These inputs preserve lineage from Phase 2 research triage into Phase 3 ruleset design.

### 3.3 Data source assumptions

The raw input dataset is the BTCUSDT Perpetual 1-minute feed generated by the Shi Aggregator.

Known source properties:

- `AggTrades` counts Binance aggTrade messages, not exact fills
- `BuyQty` / `SellQty` are taker-aggressor bar-level proxies
- liquidation data is contextual, not guaranteed complete
- synthetic candles may exist and are marked `IsSynthetic = 1`

These limitations are part of the historical truth surface and must remain visible in result interpretation.

### 3.4 Input integrity assumptions

Phase 3 assumes Analyzer inputs are already:

- UTC-normalized
- schema-validated
- deterministic on repeated runs
- ordered by timestamp
- generated without future-bar contamination under current Analyzer rules

If this assumption fails, Phase 3 outputs are invalid.

### 3.5 Inherited Analyzer constants

Phase 3 inherits the following algorithmic thresholds from Phase 1 + Phase 2 modules.
These are hardcoded in Python source — not runtime config.

| Constant | Value | Module | Semantics |
|----------|-------|--------|-----------|
| `MIN_REAL_BARS_H1` | 45 | `swings.py` | Min real 1m rows for H1 bar to participate in swing detection (out of 60) |
| `MIN_REAL_BARS_H4` | 180 | `swings.py` | Min real 1m rows for H4 bar to participate in swing detection (out of 240) |
| `MAX_STATE_GAP_MINUTES` | 3 | `failed_breaks.py` | Timestamp gap threshold — pending state resets if exceeded |
| `SETUP_TTL_BARS` | 12 | `setups.py` | Setup lifecycle window (bars from anchor to expiry) |
| `OUTCOME_HORIZON_BARS` | 12 | `outcomes.py` | Forward-looking outcome measurement window |
| `MIN_SELECTION_SCORE` | 0.05 | `selections.py` | Minimum RankingScore for SELECT decision |
| `MIN_SELECTION_SAMPLE` | 5 | `selections.py` | Minimum SampleCount for SELECT/REVIEW |
| `MIN_POSITIVE_RATE_DELTA` | 0.0 | `selections.py` | Minimum Delta_PositiveCloseReturnRate for SELECT |

Phase 3 must not silently override these values. If Phase 3 needs different thresholds
for replay purposes, it must declare them explicitly in its own ruleset contract.

### 3.6 Phase 3 interface contract

Phase 3 consumes Analyzer outputs through a declared artifact interface.

**v0.1 default interface mode:**

- Phase 3 reads pre-generated Analyzer CSV artifacts from a specified artifact directory
- Phase 3 does not call `analyzer.pipeline.run()` implicitly as part of replay

**Required interface inputs for a replay run:**

- artifact directory path
- dataset range / dataset identifier
- analyzer artifact version or generation metadata
- ruleset set identifier
- replay semantics version
- cost model version

**Reproducibility rule:**

If Phase 3 reads Analyzer CSV artifacts, it must validate expected schema presence before replay.

If a future mode is added where Phase 3 triggers Analyzer generation directly, that mode
must explicitly record Analyzer code version / git commit and artifact generation parameters
in the run manifest.

**v0.1 contract choice:**

CSV artifacts are the source-of-truth input boundary for Phase 3 replay.
Analyzer execution and Backtester execution are separate steps.

## 4. Ruleset Contract

A ruleset is the smallest fully specified backtestable strategy hypothesis.

One ruleset must be:

- deterministic
- reproducible
- self-contained

### 4.1 Ruleset unit

One row in the ruleset definition table corresponds to one ruleset candidate.

One Phase 2 candidate group may produce multiple ruleset variants, but one ruleset row always equals one fully specified replayable hypothesis.

**Recommended identifier:**

`ruleset_id = RULESET_{family}_{version}_{direction}_{ordinal}`

**Example:**

`RULESET_FB_H1_V1_SHORT_001`

### 4.2 Required ruleset fields

Each ruleset must define:

- `ruleset_id`
- `source_candidate_group`
- `source_selection_status`
- `direction`
- `setup_type`
- `eligible_event_types`
- `entry_trigger`
- `entry_timing`
- `max_entry_delay_bars`
- `invalidation_condition`
- `stop_model`
- `take_profit_model`
- `trailing_model`
- `expiry_model`
- `conflict_policy`
- `position_policy`
- `fee_model_id`
- `slippage_model_id`
- `same_bar_policy_id`
- `notes`

### 4.2.1 Entry trigger semantics

`entry_trigger` is not an arbitrary recomputation layer.

In v0.1, `entry_trigger` must be defined as a deterministic trigger expression anchored
to Analyzer-observable artifacts already materialized by Phase 2.

Allowed trigger anchors may include:

- normalized event table fields
- setup table fields
- setup lineage metadata
- shortlist / research-summary lineage identifiers

A ruleset may combine these with explicit replay-time conditions, but may not invent
a trigger that depends on information unavailable in Analyzer-observable state at the
decision point.

This contract preserves continuity between Phase 2 research candidates and Phase 3
replayable rulesets.

### 4.3 Ruleset source lineage

Every ruleset must remain traceable to the Analyzer research surface.

At minimum, lineage must preserve:

- originating setup family
- originating ranking/selection bucket
- shortlisted or research-summary source item

This prevents Phase 3 from inventing strategy logic disconnected from Phase 2.

### 4.4 v0.1 conservative rule

In v0.1, a ruleset may use only information observable at or before the moment a trade becomes active.

No future bar may participate in:

- entry qualification
- stop placement
- TP placement
- expiry calculation
- conflict resolution

> **LOCKED (v0.1):** No future bar may influence rule activation or trade management decisions.

## 5. Cost Model Contract

The cost model is a first-class contract.  
A ruleset is invalid without an explicit cost model.

### 5.1 Purpose

The cost model prevents false edges created by zero-cost replay.

A candidate may survive setup statistics and still fail after fees, slippage, and conservative fill assumptions are applied.

### 5.2 v0.1 scope

**Included:**

- entry fee
- exit fee
- entry slippage
- exit slippage
- same-bar ambiguity handling via explicit policy

**Excluded in v0.1:**

- queue position modeling
- partial fills
- market impact model
- dynamic spread model
- venue routing model

### 5.3 Required cost model fields

Each cost model must define:

- `fee_mode`
- `entry_fee_bps`
- `exit_fee_bps`
- `slippage_mode`
- `entry_slippage_bps`
- `exit_slippage_bps`
- `fill_priority_assumption`
- `same_bar_policy`
- `partial_fill_policy`
- `no_fill_policy`

### 5.4 v0.1 default policy

Default v0.1 assumptions:

- baseline implementation may run with explicit hook id `COST_MODEL_ZERO_SKELETON_ONLY`
- fee/slippage-enriched models remain supported as explicit hook extensions
- no partial fills are modeled
- no queue priority advantage is assumed
- no optimistic fill assumption is allowed

### 5.5 Same-bar ambiguity policy

If a future bar touches both stop and target after the trade becomes active, the engine must apply an explicit same-bar policy.

**v0.1 default:**

- use a conservative resolution policy
- never assume the favorable outcome without a stronger execution model

Same-bar resolution is a **versioned replay semantic** and must not be implicit.

> **LOCKED (v0.1):** No favorable same-bar outcome may be assumed by inference.

## 6. Replay Engine Contract

The replay engine is the deterministic state machine that turns rulesets and historical inputs into simulated trade lifecycles.

### 6.1 Engine responsibilities

The engine must:

- consume bars in strict timestamp order
- evaluate eligibility without lookahead
- activate entries only when allowed by replay semantics
- baseline implementation enforces signal materialization + next-bar-open activation and emits
  explicit stop/target/expiry/close evaluation events, with placeholder notes where full
  stop/target path resolution is not yet hardened
- generate one immutable trade ledger row per completed trade
- remain deterministic across repeated runs on identical inputs

### 6.2 Engine non-responsibilities

The engine must not:

- invent missing data
- interpolate hidden tick paths inside candles
- use future bars for favorable decisions
- perform live order management
- emulate exchange-specific live behavior beyond the explicit contract

### 6.3 Determinism requirement

Same dataset + same ruleset + same cost model + same replay semantics must produce identical:

- trade count
- trade ordering
- trade timestamps
- entry/exit prices under the chosen cost model
- close reasons
- summary metrics

If not, this is a bug.

## 7. Replay Semantics

This section is the core of Phase 3 Step 0.

The Backtester must not rely on implied semantics.  
All bar-level execution assumptions must be explicit.

### 7.1 Timestamp semantics

Phase 3 operates on the same 1-minute UTC bar sequence used by the Analyzer.

**v0.1 rules:**

- a signal becomes actionable only when its Analyzer confirmation/materialization timestamp is already observable
- signal evaluation is anchored to the Analyzer-confirmed timestamp, not to an earlier structural origin point
- earliest possible entry is the next bar open after the signal becomes actionable

### 7.2 Signal materialization semantics

A signal exists only when it becomes observable under the Analyzer contract.

Examples:

- confirmed swings exist only when materialized on eligible real rows
- sweeps may exist only after their source swing has been confirmed
- failed breaks confirm forward in time from prior sweep state
- `FAILED_BREAK_*` events become actionable only at `FailedBreak_*_ConfirmedTs`, not at the earlier sweep timestamp and not at the original swing timestamp

Phase 3 must preserve this observability boundary exactly as inherited from the Analyzer.
Phase 3 must inherit Analyzer confirmation semantics exactly and may not shift event
availability earlier in replay.

### 7.3 Entry timing semantics

**v0.1 default:**

- signal bar is evaluated at close
- earliest entry = next bar open
- no same-bar entry on signal-bar close
- no intrabar peeking to get a better fill than the defined entry timing allows

> **LOCKED (v0.1):** Earliest entry is next bar open.

### 7.4 Entry price convention

**v0.1 default:**

- `entry_price_raw` = next bar open
- `entry_price_effective` = `entry_price_raw` adjusted by cost model rules

Any alternative convention must be explicitly declared by the ruleset and replay semantics version.

### 7.4.1 Execution-domain limitation

**Known v0.1 limitation:**

The current research and replay input surface is derived from Binance Futures BTCUSDT bar data.
The intended future execution venue is Binance Spot Margin BTC/USDC.

Therefore, replay price path in v0.1 is a proxy for future execution conditions, not a
venue-identical fill stream. This implies that:

- spread behavior may differ
- liquidity conditions may differ
- realized fill quality on Spot Margin may differ from replay assumptions on Futures-derived bars

v0.1 may still use the current price surface for deterministic backtesting, but this venue
mismatch must remain explicitly documented in any edge interpretation.

### 7.5 Active trade semantics

Once a trade becomes active:

- stop and TP become eligible only after activation
- trailing logic starts only after its declared activation rule is satisfied
- expiry counting begins only after activation

### 7.6 Intrabar collision semantics

If both stop and target lie inside the same future bar range after activation, the engine must use the declared same-bar ambiguity policy.

**v0.1 default:**

- conservative resolution
- never assume the favorable outcome without a stronger execution model

### 7.7 Expiry semantics

Each ruleset must declare:

- whether expiry is bar-based or time-based
- the start point of expiry counting
- the close price convention on expiry

**v0.1 default:**

- expiry is counted in bars after activation
- expiry close uses the relevant bar close unless a stricter rule is declared

### 7.8 Synthetic candle semantics

Synthetic candles are part of the historical record and may not be silently deleted.

A ruleset may explicitly declare special handling for `IsSynthetic = 1`, for example:

- allow execution
- disallow new entries
- allow exit processing but not new activation

If no special handling is declared, the default ruleset policy applies.

### 7.9 Retrospective sweep bias note

The current Spec documents retrospective sweep bias: some sweep events are only known because their reference swing becomes confirmed later.

This does not invalidate Phase 3, but it remains a known bias source in edge claims.

Phase 3 must support tagging or segmenting results by event class or eligibility regime where needed.

## 9. Metrics Contract

Phase 3 measures trade-level performance, not only setup-level outcomes.

### 9.1 Required v0.1 metric groups

Implemented baseline metrics:

#### Trade count metrics
- total trades
- long trades
- short trades
- wins
- losses
- expired trades
- invalidated trades

#### Return metrics (conditional in baseline)
- expectancy/payoff/win/loss stats are computed only when explicit return-like ledger fields exist
- if no explicit return/PnL field exists, return metrics are marked as omitted in notes

#### R-based metrics
- reserved for hardening phase once stable R-contract columns are materialized in ledger

#### Path metrics
- average MAE
- average MFE
- average holding bars
- median holding bars
- close reason distribution

#### Risk / curve metrics
- equity curve
- max drawdown
- drawdown duration

Baseline caveat: equity/drawdown are non-monetary by default (`RESOLVED_TRADE_COUNT` basis).

### 9.2 Extended metrics (allowed after v0.1 core)

Acceptable later extensions:

- cumulative R curve
- rolling expectancy
- recovery factor
- richer MAE/MFE distribution reports

### 9.3 Cost-aware requirement

Every metric used for acceptance or promotion must be computed on **net** results after the cost model.

Gross metrics may be reported, but may not be used as the final acceptance surface.

> **LOCKED:** Acceptance and promotion rely on net, cost-aware metrics only.

## 10. Validation Contract

Validation answers:

**Does this ruleset meet minimum statistical acceptance criteria on the tested sample?**

This section is intentionally separate from robustness testing.

### 10.1 Scope

Validation includes:

- sample sufficiency
- expectancy after costs
- drawdown acceptability
- outlier dependence
- long/short asymmetry review
- source robustness review

### 10.2 Required validation checks

Each ruleset must be checked for:

- minimum number of completed trades (threshold TBD but mandatory)
- positive net expectancy after costs
- acceptable drawdown relative to return
- no dominant dependence on one or two extreme outliers
- acceptable asymmetry between long and short performance
- acceptable dependency on a single source bucket  
  (for example one TF, one setup subtype, one narrow context slice)

### 10.3 Validation output states

Each validation run must assign one of:

- `PASS`
- `REVIEW`
- `FAIL`

### 10.4 Threshold policy

In v0.1:

- contract structure is **LOCKED**
- numeric cutoffs may remain provisional until enough dataset evidence exists

This avoids false precision before sufficient sample accumulation.

Current implementation uses explicit provisional heuristics for sample sufficiency,
outlier dependence, long/short asymmetry, source concentration, and drawdown interpretation.

Statistical significance methodology is not mandatory for the first replay-capable
implementation, but it is not considered out of scope.

A later Phase 3 validation revision should define at least one formal significance
framework, for example:

- bootstrap confidence interval on expectancy
- permutation / shuffle-based null test
- another explicitly declared acceptance test

Until such a method is fixed, promotion claims must be interpreted conservatively.

## 11. Robustness Contract

Robustness answers:

**Does the edge generalize, or does it collapse outside the exact slice where it was found?**

This section is intentionally separate from statistical acceptance.

### 11.1 Scope

Robustness includes:

- IS/OOS split
- walk-forward evaluation
- regime segmentation
- parameter perturbation

### 11.2 Required robustness checks

Each promotable ruleset should be tested with:

- in-sample / out-of-sample split
- walk-forward or rolling-window replay
- regime split by major context state
- mild parameter perturbation around baseline settings (in baseline: external artifact surface only)

The exact walk-forward protocol is deferred in v0.1 specification details.
Before robustness hardening is considered complete, the following must be fixed explicitly:

- anchored vs sliding window mode
- train / test window sizes
- step size between windows

Current implementation uses provisional deterministic heuristics (e.g., fixed chronological split,
fixed walk-forward window count, minimum resolved-trade counts) and marks unavailable checks as
`NOT_EVALUATED`.

Regime evaluation is performed only when explicit regime labels are available.

### 11.3 Fragility criteria

A ruleset is fragile if:

- performance collapses outside one local historical segment
- one small parameter change destroys the result
- the edge disappears after simple regime separation
- OOS results diverge sharply from in-sample results

### 11.4 Output states

Each robustness run must assign one of:

- `ROBUST`
- `UNSTABLE`
- `FRAGILE`

Data splitting is a first-class Phase 3 design decision.
Robustness claims are invalid unless the split strategy is explicit.

### 11A.1 Scope

The split strategy defines how historical data is partitioned for:

- in-sample evaluation
- out-of-sample evaluation
- walk-forward analysis
- regime-level comparison

### 11A.2 v0.1 status

In v0.1, the split framework is required, but exact split parameters may remain deferred.

At minimum, the following must be fixed before robustness implementation is considered complete:

- split mode
- window sizes
- window ordering
- overlap policy
- step size

### 11A.3 Deferred design questions

Deferred but mandatory decisions:

- fixed split vs expanding window vs rolling window
- exact IS/OOS proportions or durations
- monthly vs calendar-block segmentation policy
- overlap policy between evaluation windows

## 12. Promotion Contract

Promotion answers:

**Is this ruleset eligible to move from historical validation toward execution design?**

### 12.1 Promotion states

Each ruleset must receive one of:

- `PROMOTE`
- `REVIEW`
- `REJECT`

### 12.2 Promotion prerequisites

A ruleset may be promoted only if all are true:

- replay is deterministic
- no lookahead violation is detected
- sample size is sufficient
- net expectancy after costs is positive
- drawdown is acceptable
- outlier dependence is not excessive
- robustness result is not `FRAGILE`
- ruleset contract is complete and reproducible

Promotion remains provisional until validation and robustness methodology versions are
explicitly recorded in the run manifest.

### 12.3 Interpretation boundary

`PROMOTE / REVIEW / REJECT` in Phase 3 are research-system decisions for design progression only.

They do **not** auto-authorize live execution.

Implemented seam: if validation scope exists without matching robustness scope,
promotion keeps deterministic non-promote behavior via `REVIEW` and may emit
`robustness_status = "MISSING"` for that scope.

### 12.4 v0.1 threshold policy

The promotion framework is **LOCKED** in v0.1.  
Exact numeric thresholds may remain provisional until enough data exists.

## 15. Determinism & Anti-Lookahead Guarantees

This section is **LOCKED**.

Phase 3 must preserve the spirit and mechanics of the Analyzer safety model.

### 15.1 Determinism

Same dataset + same ruleset + same cost model + same replay semantics must produce identical results.

### 15.2 No lookahead

No rule may depend on data unavailable at the moment of decision.

This applies to:

- entry
- stop placement
- TP placement
- expiry
- same-bar resolution
- trailing activation
- promotion metrics if based on replay state

### 15.3 Analyzer inheritance

The current Analyzer already enforces structural safety constraints such as:

- delayed swing confirmation
- real-row materialization discipline
- forward-only failed-break logic
- deterministic event ordering
- gap safety resets for stateful logic

Phase 3 must not break these guarantees when converting signals into trades.

### 15.4 Bias acknowledgment

Known semantic bias documented in the current Spec must remain visible in Phase 3 reporting.

At minimum, retrospective sweep bias must be treated as a documented caveat and must not be hidden behind aggregate performance numbers.

## 18. Strategic Interpretation

Phase 2 asks:

**What looks promising?**

Phase 3 asks:

**Does it still work when turned into explicit rules, replayed honestly, and measured after costs?**

That is the only valid bridge from research to execution design.
