## Крок 1 — Запустити research_cycle.py на сервері

Один виклик замість ручного probe/replay/record:

```bash
ssh server "cd /opt/aitrader && source .venv/bin/activate && python3 research_cycle.py"
```

Скрипт автоматично:
- знаходить unprocessed runs (_processed.json відсутній)
- probe кожного (setups, shortlist, FormalizationEligible)
- replay через backtester якщо FE > 0 (заморожені параметри)
- читає promotion decisions
- пише _processed.json маркери
- запускає slice analysis
- виводить JSON на stdout

Для тестового запуску без replay:
```bash
ssh server "cd /opt/aitrader && source .venv/bin/activate && python3 research_cycle.py --dry-run"
```

Збережи повний JSON output — він потрібен для всіх наступних кроків.

## Крок 2 — Валідація якості даних

Перевір секцію `diagnostics` в JSON output:

- `feed_coverage.missing_days` — чи є пропуски у feed?
- `analyzer_coverage` — скільки runs total/processed/unprocessed?
- `unprocessed_anomalies` — чи є runs з missing/empty артефактами?
- `data_quality_flags` — патерни (consecutive FE=0 тощо)

Якщо є anomalies — зафіксуй в handoff, не ігноруй.

## Крок 3 — Перевірити ключовий project signal

З JSON output перевір `results[].promotion_outcome`:
- Чи є хоч один != REJECT?
- Якщо ні — all-REJECT, research mode continues

## Крок 4 — Порівняти slice analysis з попереднім snapshot

Slice output є в JSON (`slice_output`).
Порівняти з попереднім snapshot:
- dir: `research/results/`
- naming: `reclaim_context_asymmetry_summary_<YYYY-MM-DD>.csv`
- baseline (без дати): `reclaim_context_asymmetry_summary.csv`
- знайти файл з найсвіжішою датою або baseline, якщо dated файлів ще немає

Якщо результати суттєво змінились (n, pos_rate, mean_ret дельта > noise) —
зафіксувати в новому файлі:
`research/results/reclaim_context_asymmetry_summary_<YYYY-MM-DD>.csv`

## Крок 5 — Оновити run_log.csv

З JSON output `results[]` оновити `research/run_log.csv`:
- Якщо рядок з `routine_status=UNPROCESSED` вже є → **replace**
- Якщо рядка немає → **append**
- Ніколи не дублювати рядки для одного `analyzer_run_id`

Формат рядка:
```
YYYY-MM-DD,<run_id>,<backtest_output_dir>,<FE>,<promotion_outcome>,<routine_status>,<notes>
```

## Крок 6 — Сформувати handoff для архітектора

Очисти `research/handoff/` (якщо є залишки) і запиши:

### cycle_meta.json
```json
{
  "cycle_date": "YYYY-MM-DD",
  "runs_processed": N,
  "runs_backtested": N,
  "runs_no_replay": N,
  "scope": "MM-DD → MM-DD",
  "agent_commit": "<git commit hash від кроку 7>"
}
```

### probe_summary.csv
З JSON `results[].probe`:
```
run_id,setups,shortlist_rows,formalization_eligible,replay_status,promotion_outcome
```

### promotion_details.csv
З JSON `results[].promotion_details` (тільки backtested runs):
```
run_id,derived_run,scope,promotion_decision,validation_status,robustness_status
```

### slice_comparison.csv
Рядки з попереднього snapshot + рядки з нового, з колонкою `snapshot_date`:
```
snapshot_date,direction,slice,n,pos_rate_pct,mean_ret_pct,median_ret_pct,mean_mfe_pct,mean_mae_pct
```

### slice_raw_output.txt
З JSON `slice_output`.

### previous_verdict.md
Копія останнього файлу з `research/verdicts/` (файл з найсвіжішою датою).
Якщо verdicts/ порожній — пропустити, написати в cycle_meta.json: `"previous_verdict": "none"`.

### diagnostics.json
З JSON `diagnostics` — повна секція діагностики як є.

**Handoff НЕ комітиться в git** — він тимчасовий.
