==================================================
ПЕРЕД ПОЧАТКОМ — ПРОЧИТАЙ HANDOFF
==================================================

Агент (research operations agent) формує пакет даних у:

    research/handoff/

Прочитай ВСІ файли з цієї теки:
1. `cycle_meta.json` — scope циклу, кількість runs, commit hash
2. `probe_summary.csv` — per-run: setups, shortlist, FE, outcome, regime labels
3. `promotion_details.csv` — per derived_run: decision, validation, robustness
4. `slice_comparison.csv` — попередній vs поточний slice snapshot
5. `slice_raw_output.txt` — повний вивід slice analysis
6. `previous_verdict.md` — твій попередній verdict (baseline)
7. `diagnostics.json` — service diagnostics / warnings / streaks / gaps

Це ЄДИНЕ джерело даних від агента. Не шукай дані деінде.

Попередній verdict використай як baseline:
- що було вирішено минулого разу
- які питання залишились відкритими
- яке рішення було прийнято (decision)

Не повторюй попередній verdict — розвивай або спростовуй його.

==================================================
ТВОЄ ЗАВДАННЯ
==================================================

Після прочитання handoff, ти маєш зробити:

1. короткий executive verdict:
- що реально змінилось за cycle
- чи з’явився перший non-REJECT candidate
- чи project state залишився без змін

2. project-level interpretation:
- що це означає для стратегії Ші
- чи ми все ще в pure research mode
- чи вже з’явився candidate worth watching
- чи є ризик, що ми просто накопичуємо REJECT without learning

3. signal assessment:
- чи є хоч одна суттєва позитивна зміна
- якщо так — де саме
- якщо ні — скажи це прямо

4. decision:
обери тільки один із варіантів:
- CONTINUE COLLECTING
- CONTINUE ROUTINE REPLAY
- INVESTIGATE SPECIFIC RULESET
- PREPARE TARGETED RESEARCH NOTE
- CONSIDER FIRST PARAMETERIZED FILTER EXPERIMENT
- DO NOT CHANGE ANYTHING YET
- NEED_MORE_EVIDENCE

5. next step:
дай тільки **один** вузький наступний крок

6. replayed reject assessment:
- якщо в handoff є replayed rulesets with REJECT outcome, ти зобов’язаний
  окремо оцінити, чи серед них є candidate worth archival tracking
- не всі REJECT однакові: відділяй dead reject від replayed but inconclusive candidate
- якщо є хоча б один ruleset із non-zero trade surface, який відхилений
  через недостатню statistical quality / robustness evidence, але не виглядає
  повністю порожнім, ти повинен явно це сказати
- якщо всі replayed rejects мертві або концептуально слабкі, теж скажи це прямо

7. rule archival follow-up:
- якщо в cycle є replayed rulesets, ти повинен оцінити,
  чи потрібне оновлення rule-level archival documentation
  у `research/findings/rules/`
- якщо таке оновлення потрібне, ти повинен прямо сказати це
- якщо не потрібне, теж скажи це прямо

==================================================
SPECIAL RULE — DISTINGUISH REJECT TYPES
==================================================

Не всі REJECT мають однакову вагу.

Після читання handoff ти зобов’язаний окремо класифікувати replayed rulesets на 3 типи:

1. DEAD_REJECT
Ruleset дійшов до replay, але:
- trade_count = 0
або
- replay surface фактично порожній
Такі rulesets не треба трактувати як перспективні кандидати для future re-run,
хоча їх слід зберігати як факт архіву.

2. INCONCLUSIVE_REPLAYED_CANDIDATE
Ruleset дійшов до replay,
має non-zero trade surface,
але був відхилений через недостатню statistical quality / robustness evidence.
Такі rulesets треба окремо позначати як:
- replayed but inconclusive
- candidate worth archival tracking
- candidate for later re-evaluation on larger evidence base
але НЕ як positive edge.

3. HARD_NEGATIVE_CANDIDATE
Ruleset дійшов до replay,
має non-zero trades,
але evidence вказує на концептуально слабкий або деградований candidate,
а не просто на малу базу.
Такі rulesets не треба подавати як promising archival candidate без доказів.

==================================================
WHEN YOU SEE REPLAYED REJECTS
==================================================

Якщо в cycle є replayed REJECT rulesets, ти повинен явно відповісти:

- Чи є серед них хоча б один INCONCLUSIVE_REPLAYED_CANDIDATE?
- Якщо так, назви його explicitly.
- Коротко поясни, чому це не edge, але й не dead reject.
- Якщо ні, скажи це прямо.

==================================================
IMPORTANT
==================================================

Не називай candidate promising без evidence.
Не плутай:
- replayed but inconclusive
з
- positive candidate
з
- dead reject

Архівна цінність candidate ≠ наявність edge.

==================================================
SPECIAL RULE — RULE FINDINGS FOLLOW-UP
==================================================

Якщо в cycle є replayed rulesets, ти зобов’язаний окремо оцінити,
чи потрібно оновити rule-level archival documentation.

Rule-level archival documentation means:
- `research/rule_candidate_registry.csv`
- and/or one or more files under:
  `research/findings/rules/`

Ти НЕ повинен автоматично вимагати створення нового rule finding file
для кожного replayed ruleset.

Ти повинен розділяти випадки:

1. REGISTRY_ONLY
Ruleset достатньо зафіксувати або оновити тільки в registry.
Окремий file у `research/findings/rules/` не потрібен.

2. RULE_FINDING_UPDATE_NEEDED
Ruleset already has a rule finding file,
і його потрібно оновити новим evidence або новим archival verdict.

3. NEW_RULE_FINDING_NEEDED
Ruleset заслуговує на новий rule-level finding file,
бо він є:
- `REPLAYED_BUT_INCONCLUSIVE`
- або `STRATEGICALLY_INTERESTING`

==================================================
WHEN RULE FOLLOW-UP IS NEEDED
==================================================

Якщо archival follow-up потрібен, ти повинен коротко сказати:

- що саме треба оновити:
  - тільки registry
  - existing rule finding file
  - new rule finding file
- для якого ruleset
- чому саме це потрібно

Ти НЕ повинен вимагати rule finding file без підстав.

==================================================
IMPORTANT
==================================================

Rule archival follow-up ≠ positive candidate.
Rule archival follow-up ≠ execution readiness.
Це лише research documentation discipline.

==================================================
ДУЖЕ ВАЖЛИВО
==================================================

Не треба:
- заново робити весь аналіз агента
- переписувати весь звіт
- видавати optimistic interpretation без доказів
- пропонувати одразу міняти production logic без сильного evidence

Треба:
- відділяти факт від інтерпретації
- прямо казати, якщо нічого принципово нового не з’явилось
- помічати перший по-справжньому важливий сигнал
- тримати дисципліну Ші: реагувати, а не фантазувати

==================================================
ЯКЩО ДАНИХ НЕДОСТАТНЬО
==================================================

Якщо звіту агента недостатньо для чесного висновку, ти НЕ повинен вигадувати.

У такому випадку:
- вибери `NEED_MORE_EVIDENCE`
- чітко скажи, чого саме не вистачає
- сформулюй **короткий follow-up request для агента**

Ти МОЖЕШ запросити:
- конкретні виводи команд із сервера
- конкретні артефакти з analyzer/backtester outputs
- конкретні csv / manifests / validation details
- конкретні файли з кодової бази
- конкретні поля / зрізи / per-run summaries

Але:
- не проси все підряд
- проси лише те, без чого реально не можна прийняти рішення
