# Feed Timestamp Semantics Audit - 2026-05-04

Scope: audit timestamp semantics for raw feed generated by `binance_aggregator_shi.py` and the Analyzer H1/H4 resample assumptions. This audit does not change strategy thresholds, replay policy, execution behavior, or promotion semantics.

## Findings

1. **Feed timezone: `UTC`.** `binance_aggregator_shi.py` uses `datetime.datetime.utcnow()` for both CSV row labels and daily file paths. The prior `FIXED_UTC_PLUS_2` hypothesis is not supported for feeds generated by this aggregator.
2. **Raw feed timestamp label: `CLOSE`.** `flush_candle()` writes the current UTC minute at flush time after buffering the preceding interval. Therefore raw `Timestamp=12:01` represents the one-minute candle opened at `12:00`.
3. **Analyzer assumption before this patch: open-labeled `Timestamp`.** Loader parsed `Timestamp` as UTC, and H1/H4 structure used `df.set_index("Timestamp").resample(..., label="left", closed="left")`. That means close-labeled raw rows could be bucketed as if they were open-labeled.
4. **Normalization added.** Analyzer now preserves raw labels as `FeedTimestampUTC`, derives `CandleOpenTsUTC = FeedTimestampUTC - 1 minute`, and uses that normalized open timestamp as canonical `Timestamp`.
5. **H4 detector boundary.** New H4 detector work should use `H4_BARS_UTC` with bucket boundaries `00:00 / 04:00 / 08:00 / 12:00 / 16:00 / 20:00 UTC`. Do not use a UTC+2 variant unless a different verified aggregator or postprocessor is found.
